📖 專案概述

什麼是 FlaUI BDD 測試框架？

本專案是一個基於 SpecFlow (BDD) + FlaUI (UI自動化) + NUnit (測試框架) 的 Windows 桌面應用程式自動化測試解決方案。

🏗️ 專案架構

Project_FlaUIBDD/ ├── 📁 Testcase_shopping_cart_FlaUI_BDD/ # 主要測試專案 │ ├── 📂 Features/ # 測試場景 (Gherkin格式) │ │ └── ShoppingCart.feature # 購物車功能測試場景 │ ├── 📂 StepDefinitions/ # 步驟定義實作 │ │ └── ShoppingCartSteps.cs # C# 測試步驟實作 │ ├── 📂 PageObjects/ # 頁面物件模型 │ │ ├── BasePage.cs # 基礎頁面類別 │ │ ├── ProductListPage.cs # 產品列表頁面 │ │ └── ShoppingCartPage.cs # 購物車頁面 │ ├── 📂 Helpers/ # 輔助工具 │ ├── 📂 Hooks/ # 測試生命週期鉤子 │ └── 📂 reports/ # 測試報告輸出目錄 ├── 📂 web_dashboard/ # Web 控制台 ├── 🔧 run-tests-html.ps1 # 執行測試腳本 └── 🔧 start-dashboard.ps1 # 啟動控制台腳本

➕ 如何新增測試案例

提示： 新增測試案例需要修改兩個主要檔案：Feature 檔案（測試場景）和 StepDefinitions 檔案（步驟實作）

1 編寫 Gherkin 測試場景

📁 位置：Features/ShoppingCart.feature

在 ShoppingCart.feature 檔案中新增測試場景，使用 Gherkin 語法：

@ShoppingCart @YourTag Scenario: TC08 - 您的測試案例名稱 Given 購物車已清空 When 我點擊加入蘋果按鈕 And 我點擊蘋果的增加數量按鈕 Then 蘋果數量應該是 "2" And 購物車總計應該是 "NT$ 60"

🔤 Gherkin 語法說明

關鍵字 用途 範例
Scenario: 定義一個測試場景 Scenario: TC08 - 測試商品數量
Given 前置條件 Given 購物車已清空
When 執行動作 When 我點擊加入蘋果按鈕
Then 驗證結果 Then 購物車件數應該是 "1 件"
And 連接多個步驟 And 我點擊香蕉按鈕
@標籤 分類和篩選測試 @ShoppingCart @AddItem

📝 參數化測試 (Scenario Outline)

當需要使用不同資料執行相同測試邏輯時，使用 Scenario Outline：

Scenario Outline: TC09 - 驗證商品價格 Given 購物車已清空 When 我點擊加入<商品>按鈕 Then 購物車總計應該是 "<總計>" Examples: | 商品 | 總計 | | 蘋果 | NT$ 30 | | 香蕉 | NT$ 20 | | 牛奶 | NT$ 55 |

2 實作步驟定義 (Step Definitions)

📁 位置：StepDefinitions/ShoppingCartSteps.cs

為新的測試步驟添加 C# 實作：

// 新增 When 步驟 [When(@"我點擊蘋果的增加數量按鈕")] public void When我點擊蘋果的增加數量按鈕() { _shoppingCartPage.IncreaseItemQuantity("蘋果"); TestContext.WriteLine("已增加蘋果數量"); } // 新增 Then 驗證步驟 [Then(@"蘋果數量應該是 ""(.*)""")] public void Then蘋果數量應該是(string expectedQuantity) { var actualQuantity = _shoppingCartPage.GetItemQuantity("蘋果"); ClassicAssert.AreEqual( expectedQuantity, actualQuantity, $"蘋果數量不符: 預期 {expectedQuantity}, 實際 {actualQuantity}" ); }

3 更新 Page Object（如需要）

如果新的測試步驟需要新的頁面操作，更新對應的 Page Object：

📁 位置：PageObjects/ShoppingCartPage.cs

public void IncreaseItemQuantity(string itemName) { var increaseButton = FindElement($"increase-{itemName}"); increaseButton?.AsButton().Click(); Thread.Sleep(300); // 等待 UI 更新 } public string GetItemQuantity(string itemName) { var quantityElement = FindElement($"quantity-{itemName}"); return quantityElement?.Name ?? "0"; }

4 建置專案

cd Testcase_shopping_cart_FlaUI_BDD dotnet build
⚠️ 注意： 修改 Feature 檔案後，SpecFlow 會自動重新生成 *.feature.cs 檔案。請勿手動編輯這些檔案！

▶️ 執行測試

💡 提示： FlaUI 測試必須在 Windows 環境執行。測試會啟動實際的應用程式視窗進行 UI 自動化測試。

🚀 方法一：使用 PowerShell 腳本（推薦）

執行測試並生成 HTML 報告

# 基本執行 .\run-tests-html.ps1 # 執行完成後自動開啟報告 .\run-tests-html.ps1 -OpenReport # 只執行特定標籤的測試 .\run-tests-html.ps1 -Filter "TestCategory=AddItem" # 使用 Release 配置執行 .\run-tests-html.ps1 -Configuration Release -OpenReport

✅ 腳本會自動執行以下步驟：

  1. 啟動 Demo 購物車網頁 (localhost:8888)
  2. 還原 NuGet 套件
  3. 建置測試專案
  4. 執行測試並生成 HTML 報告
  5. 顯示測試結果摘要

⚙️ 方法二：使用 .NET CLI 命令

步驟 1：啟動測試應用程式

# 開啟新的 PowerShell 視窗執行 cd demo/shopping_cart python -m http.server 8888

步驟 2：執行測試

# 進入測試專案目錄 cd Testcase_shopping_cart_FlaUI_BDD # 還原套件（首次執行） dotnet restore # 建置專案 dotnet build # 執行所有測試 dotnet test # 執行特定測試 dotnet test --filter "TestCategory=ShoppingCart" dotnet test --filter "TestCategory=AddItem" dotnet test --filter "FullyQualifiedName~TC01"

✏️ 修改測試案例

⚠️ 重要： 修改測試案例時，務必確保 Feature 檔案和 Step Definitions 保持同步！

📝 修改現有測試場景

情境 1：修改測試步驟

步驟 1： 編輯 Feature 檔案

# 修改前 When 我點擊加入蘋果按鈕 Then 購物車件數應該是 "1 件" # 修改後（增加更多驗證） When 我點擊加入蘋果按鈕 Then 購物車件數應該是 "1 件" And 蘋果應該顯示在購物車中 And 購物車圖示應該顯示數量標記

步驟 2： 如果使用了新的步驟，在 StepDefinitions 中實作

[Then(@"蘋果應該顯示在購物車中")] public void Then蘋果應該顯示在購物車中() { var isDisplayed = _shoppingCartPage.IsItemInCart("蘋果"); ClassicAssert.IsTrue( isDisplayed, "蘋果未顯示在購物車中" ); }

情境 2：修改驗證數值

# 修改前 Then 購物車總計應該是 "NT$ 30" # 修改後（價格變更） Then 購物車總計應該是 "NT$ 35"

💡 提示： 只需修改 Feature 檔案中的參數值，不需要修改 Step Definitions！

情境 3：修改測試標籤

# 修改前 @ShoppingCart @AddItem Scenario: TC01 - 加入單一商品 # 修改後（增加優先級標籤） @ShoppingCart @AddItem @Priority1 @Smoke Scenario: TC01 - 加入單一商品

🔄 修改 Page Object

何時需要修改 Page Object？

  • ✏️ UI 元素的 AutomationId 改變
  • ➕ 新增頁面元素或操作
  • 🔧 修改元素定位策略
  • ⚡ 優化等待時間或錯誤處理

範例：修改元素定位

📁 PageObjects/ProductListPage.cs

// 修改前 public void ClickAddApple() { var button = FindElement("add-apple"); button?.AsButton().Click(); } // 修改後（使用更靈活的定位） public void AddItem(string itemName) { var testId = $"add-{itemName.ToLower()}"; var button = FindElement(testId); if (button != null) { button.AsButton().Click(); Thread.Sleep(500); // 增加等待時間 } else { throw new ElementNotFoundException( $"找不到 {itemName} 的加入按鈕" ); } }

⚙️ 修改測試配置

📁 位置：App.config

<appSettings> <!-- 修改應用程式路徑 --> <add key="ApplicationPath" value="C:\Path\To\Your\App.exe" /> <!-- 修改應用程式 URL --> <add key="ApplicationUrl" value="http://localhost:8888" /> <!-- 調整隱式等待時間（毫秒）--> <add key="ImplicitWaitTimeout" value="10000" /> </appSettings>

✅ 修改後的檢查清單

修改完成後，請確認：

  • ☑️ Feature 檔案語法正確（無拼寫錯誤）
  • ☑️ 所有步驟都有對應的 Step Definition
  • ☑️ Page Object 方法已更新（如需要）
  • ☑️ 專案建置成功（無編譯錯誤）
  • ☑️ 執行測試驗證修改正確

驗證修改

# 建置專案 dotnet build # 執行特定測試 dotnet test --filter "FullyQualifiedName~TC01" # 查看測試報告確認

🔧 疑難排解

❌ 常見問題與解決方案

問題 1: 找不到應用程式視窗

錯誤訊息： "Could not find application window"

解決方案：

  1. 確認 App.config 中的 ApplicationPath 正確
  2. 確認應用程式已啟動（檢查 http://localhost:8888）
  3. 檢查 WebView2 是否正確安裝
  4. 增加 ImplicitWaitTimeout 等待時間

問題 2: 元素找不到

錯誤訊息： "Element with AutomationId 'xxx' not found"

解決方案：

  1. 使用 Inspect.exe 工具驗證 AutomationId
  2. 增加等待時間讓頁面完全載入
  3. 檢查元素定位策略是否正確
  4. 確認應用程式 UI 沒有改變

問題 3: SpecFlow 步驟未定義

錯誤訊息： "No matching step definition found"

解決方案：

  1. 檢查 Feature 檔案中的步驟文字
  2. 確認 StepDefinitions 中有對應的 [Given/When/Then] 方法
  3. 檢查正規表達式是否匹配
  4. 重新建置專案 dotnet build

問題 4: NuGet 套件還原失敗

錯誤訊息： "Package restore failed"

解決方案：

# 清除 NuGet 快取 dotnet nuget locals all --clear # 重新還原 dotnet restore --force

問題 5: 測試執行過快導致失敗

症狀： 測試隨機失敗，重新執行又通過

解決方案：

  1. 在 BasePage.cs 中增加 WaitForElement 方法
  2. 在關鍵操作後增加 Thread.Sleep()
  3. 使用顯式等待而非固定延遲
  4. 調整 App.config 中的等待時間

💡 最佳實踐建議

  1. 保持測試獨立性 - 每個測試都應該可以獨立執行
  2. 使用有意義的測試名稱 - 清楚描述測試目的
  3. 善用標籤 - 使用 @標籤 分類和組織測試
  4. 定期執行測試 - 建立 CI/CD 自動化流程
  5. 維護 Page Object - 保持 POM 結構清晰
  6. 記錄測試結果 - 保存歷史報告追蹤品質趨勢
  7. 編寫清晰的 Gherkin - 讓非技術人員也能理解
  8. 適度使用等待 - 避免過度使用 Thread.Sleep()
